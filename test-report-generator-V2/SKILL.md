# Test Report Generator V2

## Overview

生成 需求名+版本+测试报告 Word文档，样式适合直接复制到邮件中发送。

## Report Structure

```
前序: 需求描述 + 需求链接 (必须)
一、测试结论 (必须)
   1、功能测试：描述，链接
   2、效果测试：描述
   3、性能测试：描述
   4、稳定性测试：描述
      1) 子项目1
      2) 子项目2
   
   问题单修复情况：(表格)
   | 问题单 | 问题描述 | 状态 |
   
   遗留问题：
   1.问题描述，问题单：链接
   
二、详细数据 (必须,但子章节动态显示)
   - 2.1 效果测试数据 (如有)
   - 2.2 性能测试数据 (如有)
   - 2.3 稳定性数据 (如有)
   - 2.4 旁路测试数据 (如有)
```

**规则**: 
- 前序和测试结论章节必须存在
- 测试结论使用列表形式(1、2、3...)
- 问题单表格和遗留问题根据数据显示
- 详细数据的子章节根据用户提供的数据动态显示
- 未提供数据的章节不显示

## Workflow

### Step 1: Collect User Input

向用户询问以下信息:

**必须信息:**
1. 需求描述 (一句话描述本次测试的需求)
2. 需求链接 (TAPD链接等)
3. 测试结论列表 (每种测试类型的结论)

**可选信息:**
4. 问题单列表 (bug修复情况)
5. 遗留问题 (未修复的问题+链接)
6. 效果测试数据 (数据描述、链接、图片)
7. 性能测试数据 (数据描述、链接、图片)
8. 稳定性测试数据 (数据描述、链接、图片)
9. 旁路测试数据 (数据描述、链接、图片)

## Key Methods

| Method | Description | Required |
|--------|-------------|----------|
| `add_preface(requirement_desc, requirement_link)` | 添加前序(需求描述+链接) | Yes |
| `add_conclusion(items, bug_table, remaining_issues)` | 添加测试结论(新格式) | Yes |
| `add_conclusion(summary, problems)` | 添加测试结论(旧格式,兼容) | Yes |
| `add_effect_data(...)` | 添加效果测试数据 | No |
| `add_performance_data(...)` | 添加性能测试数据 | No |
| `add_stability_data(...)` | 添加稳定性测试数据 | No |
| `add_bypass_data(...)` | 添加旁路测试数据 | No |
| `save(output_path)` | 保存报告 | Yes |

## Style Guide

1. **简洁**: 无花哨装饰,适合邮件
2. **表格**: 使用简单边框表格
3. **链接**: 蓝色文字显示
4. **标题**: 使用中文数字编号 (一、二、2.1、2.2)
5. **列表**: 测试结论使用阿拉伯数字编号 (1、2、3)
6. **字体**: 微软雅黑,正文10.5pt

## AI Instructions

当用户请求生成测试报告时:

### Step 0: 获取需求详情 (重要)

**如果用户提供了TAPD需求链接，必须先通过MCP获取需求详情：**

1. 从链接中解析 `workspace_id` 和 `story_id`
   - 链接格式: `https://tapd.woa.com/tapd_fe/{workspace_id}/story/detail/{story_id}`
   - 示例: `https://tapd.woa.com/tapd_fe/10163841/story/detail/1010163841128500102`
     - workspace_id = 10163841
     - story_id = 1010163841128500102

2. 调用 TAPD MCP 的 `stories_get` 工具获取需求详情:
   ```
   stories_get(workspace_id=10163841, id="1010163841128500102")
   ```

3. 从返回结果中提取需求信息，生成需求描述:
   - 需求名称 (name)
   - 需求描述摘要 (description 的前几句话)
   - 组合成: "{需求名称}。{描述摘要}"

### Step 1: 收集用户输入

**必须信息:**
1. 需求链接 (TAPD链接) - 用于自动获取需求详情
2. 测试结论列表 (每种测试类型的结论)

**可选信息:**
3. 问题单列表 (bug修复情况) - 可通过MCP的 `get_stories_related_bugs` 获取
4. 遗留问题 (未修复的问题+链接)
5. 效果/性能/稳定性/旁路测试数据

### Step 2: 获取关联Bug (如需要)

如果用户提供了bug链接或需要显示问题单：

1. 调用 `get_stories_related_bugs` 获取需求关联的bug ID列表
2. 调用 `bugs_get` 获取bug详情 (title, status等)
3. 整理成 bug_table 格式

### Step 3: 生成报告

1. **整理结论**: 将各类测试结果整理为列表形式(items)
2. **问题单**: 整理bug修复情况(bug_table)和遗留问题(remaining_issues)
3. **生成描述**: 根据用户提供的数据,AI生成数据描述文本
4. **整理数据**: 将用户提供的表格数据整理成统一格式
5. **生成文件名**: 按照命名规则生成文件名
6. **生成报告**: 调用脚本生成docx文件
7. **确认输出**: 告知用户报告已生成,可直接复制到邮件

### 文件命名规则 (重要)

**格式**: `{需求名简介}_{版本号}_测试报告.docx`

**生成规则**:
1. 从需求名称中提取关键词作为简介(去掉【】标签等)
2. 从需求名称或描述中提取版本号(如V5.3.4、V6.0.0等)
3. 组合成文件名

**示例**:
- 需求名: "【文字识别】【票据】耗时优化方案更新V5.3.4"
- 生成文件名: `票据耗时优化_V5.3.4_测试报告.docx`

- 需求名: "二维码识别功能V6.0.0版本"
- 生成文件名: `二维码识别功能_V6.0.0_测试报告.docx`

**注意**: 
- 文件名中不要包含特殊字符(如【】、/、\等)
- 如果需求名没有版本号，从用户提供的其他信息中获取，或询问用户

**MCP返回**:

```json
  "description": "本次需求主要优化二维码识别能力，提升端到端性能..."

```

**生成的需求描述**:

```
票据V5.3.4版本二维码功能。本次需求主要优化二维码识别能力，提升端到端性能。
```

- 链接和图片由用户提供,AI不要编造
- 数据描述由AI根据表格数据生成
- 未提供的章节不要显示
- 优先使用新格式(items列表),旧格式(summary)仍兼容

### 详细数据描述规范 (重要)

**二、详细数据章节的每个子项必须包含详细的数据客观描述，禁止使用主观评价词汇。**

**禁止使用的词汇**:
- 较好、优异、出色、良好、优秀
- 显著提升、大幅优化、明显改善
- 表现不错、效果理想
- 其他带有主观评价性质的形容词

**必须遵循的描述原则**:
1. **客观陈述数据**: 只描述具体数值、百分比、对比差异
2. **详细罗列**: 逐项说明每个指标的具体数值
3. **对比说明**: 新旧版本对比时，明确列出各自的数值和差值

**示例**:

**错误写法**:
```
性能测试结果良好，新版本性能显著提升，表现优异。
```

**正确写法**:
```
性能测试数据如下：
- V5.3.4开启二维码端到端平均耗时1.49s，V5.3.3(串行环境)为2.97s，差值1.48s，降幅49.8%
- V5.3.4开启二维码端到端平均耗时1.49s，V5.3.3(并行环境)为1.64s，差值0.15s，降幅9.1%
- 检测环节：V5.3.4耗时120ms，V5.3.3耗时150ms，降低30ms
- 分类环节：V5.3.4耗时80ms，V5.3.3耗时100ms，降低20ms
```

**旁路测试数据示例**:

**错误写法**:
```
旁路测试结果表明新版本整体表现较好，各类票据识别效果均有提升。
```

**正确写法**:
```
旁路测试共计400个样本，新旧版本对比结果：
- 第一类票据(35个)：旧版本判定好21个，新版本判定好14个，差值-7
- 第二类票据(26个)：旧版本判定好6个，新版本判定好20个，差值+14
- 第三类票据(X个)：新旧版本判定结果基本持平
- 第四类票据(41个)：旧版本判定好16个，新版本判定好25个，差值+9
- 总计：新版本在第二、四类票据上判定好的数量多于旧版本，在第一类票据上少于旧版本
```

**效果测试数据示例**:

**正确写法**:
```
效果测试数据对比(V6.0.0 vs V5.8.0)：
- 正面通过率：V6.0.0为95.2%，V5.8.0为93.1%，差值+2.1%
- 反面通过率：V6.0.0为89.5%，V5.8.0为82.3%，差值+7.2%
- 整体成功率：V6.0.0为100%，V5.8.0为100%，无变化
- 请求总数：V6.0.0测试100次，V5.8.0测试100次
```

**稳定性测试数据示例**:

**正确写法**:
```
稳定性测试运行15小时，具体数据：
- 运行时长：15小时
- 请求总数：50000次
- 成功率：99.98%（失败10次）
- 平均响应时间：52ms
- P99响应时间：120ms
- 内存占用：初始2.1G，最终4.4G，增长2.3G
- 显存占用：T4卡独占13.5G，无明显增长
```
